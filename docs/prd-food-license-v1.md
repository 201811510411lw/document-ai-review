# 食品安全证照检测 V1 PRD

本文档定义 `document-ai-review` 食品安全证照检测 V1 的产品范围、流程、职责边界和验收标准。

`README.md` 仍然是项目唯一主上下文。本文档只补充 V1 食品安全证照检测 PRD，不替代 README。

---

## 1. 背景和目标

`document-ai-review` 是企业内部 Document AI Review Agent / Skill 平台，当前 V1 聚焦 **食品安全证照检测**。

企业在供应商准入、资质复核和日常合规审核中，需要检查供应商是否提供有效的食品安全相关证照，并判断证照主体、统一社会信用代码、经营项目、有效期等信息是否满足业务要求。传统人工审核存在效率低、标准不一致、过程难追溯、复核成本高等问题。

V1 的目标不是建设单纯 OCR 工具，而是先跑通一个可验证的食品安全证照审核闭环：

- 将 OCR 文本、文件路径或上传文件中的证照信息转为结构化审核数据；
- 使用 LangGraph 显式编排食品安全证照检测流程；
- 使用 LangChain 负责模型调用、Prompt、结构化输出和工具封装；
- 使用 Python 规则引擎执行确定性规则校验；
- 基于 Python 规则引擎结果汇总最终风险等级；
- 通过 FastAPI 对外提供 HTTP API；
- 使用独立 MySQL 审核结果库保存审核任务、审核结果、规则结果、人工复核和审计日志；
- 为后续扩展其他 Document AI Review Skill 沉淀可复用的审核链路。

V1 第一阶段优先支持 **OCR 文本输入** 跑通闭环。文件上传、图片解析、PDF 解析保留接口设计和扩展空间，但实现优先级低于 OCR 文本闭环。

## 2. V1 范围

V1 采用纯 Python 架构，当前只有一个 Python 服务：`ai-service`。

V1 包含：

- FastAPI HTTP API；
- 审核任务 Service / Repository；
- LangGraph 食品安全证照检测工作流；
- LangChain 字段抽取 Chain、Prompt、结构化输出和工具封装；
- Python 规则引擎；
- Pydantic 审核请求、证照字段、规则结果和审核结果模型；
- MySQL 审核结果库数据保存；
- 人工复核记录和审计留痕。

V1 第一阶段优先实现：

- 通过 OCR 文本创建食品安全证照审核任务；
- 接收供应商名称、统一社会信用代码、供应商经营地址等业务侧信息；
- 从 OCR 文本中抽取食品安全证照字段；
- 规范化主体名称、统一社会信用代码、经营项目、日期和地址；
- 执行食品安全证照规则校验；
- 汇总规则结果并生成最终风险等级；
- 判断是否需要人工复核；
- 保存审核任务、审核结果、字段结果、规则结果和审计日志；
- 返回结构化审核结果。

V1 保留但降低优先级：

- 上传食品安全证照图片；
- 上传食品安全证照 PDF；
- 通过本地文件路径读取图片或 PDF；
- 图片解析和 PDF 解析；
- 多模态模型直接读取证照图片。

## 3. V1 不做事项

V1 不实现：

- Java / Spring Boot 服务；
- 任何 Java 模块作为 V1 实现范围；
- 双服务拆分架构；
- 独立管理后台；
- 完整权限系统；
- 企业微信、邮件、OA 待办通知；
- 第三方证照验真平台；
- RAG / 知识库；
- 合同审核、QC 证照、烟草证等其他 Skill；
- 生产级多租户和复杂组织权限；
- 对图片和 PDF 解析能力做生产级优化；
- 使用 LLM 直接给出最终规则判定。

后续企业系统仍可通过 HTTP API 调用 Python `ai-service`，但这不属于 V1 核心实现范围。

## 4. 用户角色

| 角色 | 说明 | 核心诉求 |
| --- | --- | --- |
| 业务系统调用方 | ERP、OA、测试脚本或其他内部系统 | 通过 HTTP API 提交审核任务并获取结构化审核结果 |
| 审核人员 | 负责供应商资质审核的人 | 查看风险项，对需人工复核的任务作出通过、驳回或补充材料判断 |
| 复核管理人员 | 负责审核规则和复核质量的人 | 追踪审核结果、规则命中情况和人工复核记录 |
| 开发 / 测试人员 | 建设和验证 V1 的人 | 使用 OCR 文本、本地样例和测试脚本验证最小闭环 |
| 运维人员 | 部署和维护 `ai-service` 的人 | 配置模型、数据库和运行环境，排查审核任务异常 |

## 5. 核心流程

V1 第一阶段核心流程：

```text
OCR 文本 + 供应商业务信息
    ↓
FastAPI 创建审核任务
    ↓
Service / Repository 保存任务初始状态
    ↓
LangGraph 执行食品安全证照检测工作流
    ↓
load_document 标准化 OCR 文本输入
    ↓
classify_document 判断是否为食品安全相关证照
    ↓
extract_fields 通过 LangChain 抽取结构化字段
    ↓
normalize_fields 规范化字段
    ↓
run_rules 调用 Python 规则引擎执行规则校验
    ↓
summarize_risk 基于规则结果汇总最终风险等级和摘要建议
    ↓
route_review 判断自动通过、自动驳回或进入人工复核
    ↓
Repository 保存审核结果、规则结果、人工复核状态和审计日志
    ↓
FastAPI 返回结构化审核结果
```

职责边界：

- FastAPI 负责 HTTP API，不直接承载规则判断或模型调用逻辑；
- LangGraph 负责食品安全证照检测 V1 核心流程编排；
- LangChain / LLM 只负责字段抽取、结构化输出和摘要建议；
- LangChain / LLM 不直接做最终规则判定；
- Python 规则引擎负责确定性规则校验；
- 最终风险等级由 Python 规则引擎结果汇总得到；
- MySQL 审核结果库负责保存审核任务、审核结果、规则结果、人工复核和审计日志。

## 6. LangGraph 工作流节点

| 节点 | 输入 | 输出 | 职责 |
| --- | --- | --- | --- |
| `load_document` | OCR 文本、文件路径或上传文件元信息 | 统一文档内容 | 第一阶段优先标准化 OCR 文本输入；文件、图片、PDF 作为后续扩展输入 |
| `classify_document` | 统一文档内容 | 证照类型识别结果 | 判断材料是否为食品安全相关证照 |
| `extract_fields` | 文档内容、证照类型 | 食品安全证照抽取字段 | 使用 LangChain、Prompt 和结构化输出抽取证照字段 |
| `normalize_fields` | 抽取字段 | 规范化字段 | 规范化日期、主体名称、统一社会信用代码、经营项目和地址 |
| `run_rules` | 规范化字段、供应商业务信息 | 规则结果列表 | 调用 Python 规则引擎执行确定性规则校验 |
| `summarize_risk` | 规则结果列表、规范化字段 | 风险等级、摘要建议 | 基于规则结果汇总最终风险等级，可使用 LLM 辅助生成摘要建议 |
| `route_review` | 风险等级、规则结果列表 | 审核路由结果 | 判断自动通过、自动驳回或进入人工复核 |

LangGraph state 至少包含：

- `task_id`：审核任务 ID；
- `input`：OCR 文本、文件路径或上传文件元信息；
- `supplier`：供应商名称、统一社会信用代码、经营地址等业务侧信息；
- `document`：统一文档内容；
- `document_type`：证照类型识别结果；
- `extracted_fields`：食品安全证照抽取字段；
- `normalized_fields`：规范化后的字段；
- `rule_results`：规则校验结果；
- `risk_level`：总体风险等级；
- `summary`：审核建议摘要；
- `needs_manual_review`：是否需要人工复核；
- `audit_events`：工作流执行过程中的关键事件。

## 7. 输入输出

### 7.1 输入

V1 第一阶段优先输入：

- `ocr_text`：食品安全证照 OCR 文本；
- `supplier_name`：供应商名称；
- `supplier_credit_code`：供应商统一社会信用代码；
- `supplier_address`：供应商经营地址；
- `declared_document_type`：可选，业务侧声明的证照类型。

V1 保留接口设计但降低实现优先级的输入：

- `file_path`：本地证照图片或 PDF 路径；
- `uploaded_file`：上传的图片或 PDF；
- `file_type`：图片、PDF 或其他文件类型；
- `ocr_provider_result`：外部 OCR 服务返回的结构化或半结构化结果。

### 7.2 输出

审核结果至少包含：

- `task_id`：审核任务 ID；
- `document_type`：证照类型；
- `status`：审核任务状态；
- `risk_level`：最终风险等级；
- `needs_manual_review`：是否需要人工复核；
- `extracted_fields`：模型抽取字段；
- `normalized_fields`：规范化字段；
- `rule_results`：规则结果列表；
- `summary`：审核建议摘要；
- `created_at`：任务创建时间；
- `updated_at`：任务更新时间。

示例：

```json
{
  "task_id": "review-task-001",
  "document_type": "food_license",
  "status": "REVIEWED",
  "risk_level": "HIGH",
  "needs_manual_review": true,
  "extracted_fields": {
    "subject_name": "成都示例食品有限公司",
    "credit_code": "91510100MA00000000",
    "license_no": "JY15101000000000",
    "valid_to": "2025-01-01"
  },
  "rule_results": [
    {
      "rule_code": "FOOD_LICENSE_EXPIRED",
      "rule_name": "证照是否过期",
      "passed": false,
      "risk_level": "HIGH",
      "message": "当前日期超过证照有效期截止日期"
    }
  ],
  "summary": "发现高风险问题，建议人工复核证照有效期。"
}
```

## 8. 食品安全证照字段

V1 建议抽取并规范化以下字段：

| 字段 | 说明 | 是否关键字段 |
| --- | --- | --- |
| `subject_name` | 主体名称 | 是 |
| `credit_code` | 统一社会信用代码 | 是 |
| `license_no` | 许可证编号 | 是 |
| `business_address` | 经营场所 | 否 |
| `legal_person` | 法定代表人 / 负责人 | 否 |
| `business_items` | 经营项目 | 是 |
| `valid_from` | 有效期开始日期 | 否 |
| `valid_to` | 有效期截止日期 | 是 |
| `issue_authority` | 发证机关 | 否 |
| `issue_date` | 发证日期 | 否 |

字段规范化要求：

- 日期统一规范为 `YYYY-MM-DD`；
- 统一社会信用代码去除空格并保持大写；
- 主体名称去除多余空白和明显 OCR 噪声；
- 经营项目规范为字符串列表；
- 地址去除多余空白，保留可用于相似度判断的原始语义。

## 9. 规则清单

V1 第一阶段建议实现以下规则：

| 规则编码 | 规则名称 | 风险等级 | 说明 |
| --- | --- | --- | --- |
| `FOOD_LICENSE_EXISTS` | 证照是否存在 | 高 | OCR 文本为空，或没有可审核的证照输入 |
| `FOOD_LICENSE_TYPE_MATCH` | 证照类型是否匹配 | 高 | 材料无法识别为食品安全相关证照 |
| `FOOD_LICENSE_NO_REQUIRED` | 许可证编号是否为空 | 高 | 食品经营许可证编号不能为空 |
| `FOOD_LICENSE_EXPIRED` | 证照是否过期 | 高 | 当前日期超过证照有效期截止日期 |
| `SUBJECT_NAME_MATCH` | 主体名称是否一致 | 中 | 证照主体名称与供应商名称不一致 |
| `CREDIT_CODE_MATCH` | 统一社会信用代码是否一致 | 高 | 证照信用代码与业务系统信用代码不一致 |
| `BUSINESS_SCOPE_COVERED` | 经营项目是否覆盖食品业务 | 中 | 经营项目不包含食品销售、预包装食品、散装食品等食品业务范围 |
| `ADDRESS_SIMILARITY` | 经营场所是否近似匹配 | 低 | 证照地址与业务系统地址相似度较低 |
| `REQUIRED_FIELDS_EXTRACTED` | 关键字段是否完整 | 中 | 主体名称、信用代码、许可证编号、有效期、经营项目等关键字段缺失 |

规则执行要求：

- 规则判断由 Python 规则引擎执行；
- 每条规则输出规则编码、规则名称、是否通过、风险等级和提示信息；
- 规则结果应可测试、可解释、可保存；
- LangChain / LLM 不直接决定规则是否通过；
- LLM 输出只能作为字段抽取、结构化输出和摘要建议的输入。

## 10. 风险等级

V1 风险等级由 Python 规则引擎结果汇总得到。

| 风险等级 | 说明 | 典型情况 |
| --- | --- | --- |
| `HIGH` | 高风险，影响证照有效性或主体合法性判断 | 证照缺失、类型不匹配、许可证编号为空、证照过期、信用代码不一致 |
| `MEDIUM` | 中风险，需要人工确认业务一致性或字段可靠性 | 主体名称不一致、经营项目未覆盖食品业务、关键字段缺失 |
| `LOW` | 低风险，不直接阻断审核但建议关注 | 经营场所相似度低、非关键字段异常 |
| `NONE` | 无明显风险 | 所有启用规则通过 |

汇总规则建议：

- 存在任一高风险失败规则，最终风险等级为 `HIGH`；
- 不存在高风险失败规则，但存在中风险失败规则，最终风险等级为 `MEDIUM`；
- 只存在低风险失败规则，最终风险等级为 `LOW`；
- 所有启用规则通过，最终风险等级为 `NONE`。

## 11. 人工复核触发条件

需要进入人工复核：

- 任一高风险规则失败；
- 任一中风险规则失败；
- 证照类型无法明确识别；
- 关键字段缺失或抽取结果不可用；
- OCR 文本质量不足，导致规则无法可靠执行；
- 工作流节点执行异常但任务仍可保留待人工判断；
- 审核策略要求低风险问题也进入人工复核。

可自动通过：

- 材料识别为食品安全相关证照；
- 关键字段完整；
- 高风险和中风险规则全部通过；
- 不存在需要人工确认的低风险策略；
- 最终风险等级为 `NONE` 或允许自动放行的 `LOW`。

可自动驳回：

- 明确不存在证照输入；
- 明确不是食品安全相关证照；
- 证照已过期且业务策略允许自动驳回；
- 统一社会信用代码明确不一致且业务策略允许自动驳回。

自动驳回策略在 V1 第一阶段应保持保守，默认优先进入人工复核。

## 12. 数据保存范围

V1 至少保存：

- 审核任务：任务 ID、输入来源、供应商信息、任务状态、创建时间、更新时间；
- 审核结果：文档类型、最终风险等级、审核建议、是否需要人工复核；
- 字段抽取结果：食品安全证照结构化字段；
- 字段规范化结果：规范化后的日期、主体名称、信用代码、经营项目和地址；
- 规则结果：规则编码、规则名称、是否通过、风险等级、提示信息；
- 人工复核记录：复核动作、复核人、复核备注、复核时间；
- 审计日志：任务创建、工作流开始、节点执行、规则校验、风险汇总、人工复核等关键事件。

V1 使用独立 MySQL 审核结果库作为持久化目标。自动化测试通过 mock MySQL 连接验证持久化契约，不再依赖 SQLite 作为审核结果存储。

- 可以通过 FastAPI 使用 OCR 文本创建食品安全证照审核任务；
