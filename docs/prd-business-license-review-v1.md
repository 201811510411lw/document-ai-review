# 营业执照单证审核 V1 PRD

本文档定义 `document-ai-review` 营业执照单证审核 V1 的产品范围、实现边界、测试决策和并发实施建议。

`README.md` 仍然是项目唯一主上下文。本文档只补充营业执照单证审核 tracer bullet 的实现 PRD，不替代 README，也不替代现有总需求文档。

---

## Problem Statement

当前项目已经具备 `use_cases + capabilities + workflow + ReviewResult` 的平台骨架，也已经有 `food_license` 和 `product_report` 的局部审核样板，但还没有跑通“从 MySQL SQL 获取营业执照来源记录，再根据 URL 获取文件，解析字段，执行合规规则，并输出可落库审核报告”的完整闭环。

从业务视角看，这意味着：

- 业务系统无法把营业执照审核任务稳定接入到 `ai-service`；
- 审核人员无法拿到统一、结构化、可复核的营业执照审核结果；
- 后续烟草证或多证一致性比对缺少第一条可复用的单证 tracer bullet；
- 团队虽然已经把需求拆成了 issue，但如果没有统一 PRD，多个 agent 并发开发时容易在输入契约、模块边界、测试范围上出现分叉。

本 PRD 的目标，是让营业执照成为平台里第一条真正可运行、可测试、可追溯、可扩展的单证审核链路。

## Solution

V1 先只做营业执照单证审核，不直接实现烟草证或双证一致性比对。系统应支持执行约定的 MySQL SQL，读取营业执照来源记录，将记录标准化为审核任务；基于记录中的 URL 下载营业执照文件，经过统一文档加载与文本获取链路，抽取营业执照核心字段；再通过确定性规则引擎完成合规判断，输出统一 `ReviewResult`，并保存审核结果快照。

V1 的设计原则是：

- 先做一条完整单证链路，而不是一次做多证照；
- 先保证可测试和可追溯，再谈真实 OCR、真实 LLM 或 OA 集成；
- 先沉淀稳定的输入输出契约，再扩展更多证照类型；
- 让每个实现模块都能被 `tdd` skill 独立驱动，并支持多个 agent 并发推进。

## User Stories

1. 作为业务系统调用方，我希望系统能够执行约定的 MySQL SQL 拉取营业执照记录，以便我不需要手工整理审核输入。
2. 作为业务系统调用方，我希望每条来源记录都能被标准化为统一审核任务，以便后续流程不依赖上游字段命名差异。
3. 作为开发人员，我希望来源记录能够保留主键、URL 和来源上下文，以便审核结果可以追溯到原始数据。
4. 作为审核人员，我希望系统能够根据 URL 自动获取营业执照文件，以便我不需要手工上传单张证照。
5. 作为审核人员，我希望系统能够处理 PDF 和图片等常见营业执照文件格式，以便适配常见上游来源。
6. 作为审核人员，我希望系统能够抽取营业执照的证照类型、企业名称、统一社会信用代码、经营场所或住所、法定代表人或经营者、有效期等核心字段，以便进行后续合规审核。
7. 作为审核人员，我希望系统能够判断当前材料是否真的是营业执照，以便避免错误文档进入合规判断。
8. 作为审核人员，我希望系统能够校验营业执照主体名称是否与来源主体一致，以便发现主体不匹配风险。
9. 作为审核人员，我希望系统能够校验统一社会信用代码是否缺失或异常，以便快速识别高风险材料。
10. 作为审核人员，我希望系统能够识别营业执照是否过期或临近过期，以便及时做人工复核或驳回。
11. 作为审核人员，我希望系统输出统一 `ReviewResult`，包含规则结果、风险等级、人工复核边界和审核摘要，以便我能够直接消费审核结论。
12. 作为审核人员，我希望审核结果能够落库并保留抽取字段、规则结果和来源证据快照，以便后续查询、复核和导出。
13. 作为开发人员，我希望营业执照审核实现遵循现有 `use_case -> workflow -> capability -> rules -> ReviewResult` 主链路，以便不破坏平台架构收口方向。
14. 作为开发人员，我希望关键模块都可以脱离真实 MySQL、真实 OCR、真实 LLM 独立测试，以便在本地快速迭代。
15. 作为开发人员，我希望营业执照的实现能够沉淀出后续烟草证或多证一致性比对的扩展边界，以便后续功能不是推倒重来。
16. 作为并发开发的 agent，我希望输入契约、审核结果契约和测试优先级已经明确，以便我能在不反复对齐的情况下独立完成 issue。
17. 作为项目维护者，我希望这次实现先做最小可用闭环，不包含 OA 回写、企微通知和生产级 OCR，以便控制范围和交付节奏。

## Implementation Decisions

- V1 以营业执照单证审核为唯一实现目标，不实现烟草证解析，也不实现双证一致性比对。
- 审核主链路沿用现有平台架构：由 Review Service 创建审核任务，由 use_case 承接场景入口，由 workflow 编排文档获取、字段抽取、规则执行和人工复核路由，由统一 `ReviewResult` 对外返回结果。
- MySQL 取数被定义为“来源任务获取模块”，它的职责是执行约定 SQL、读取来源记录、标准化字段、处理缺字段和空结果，不直接承担规则判断或字段抽取。
- 需要建立一个“来源记录标准化模块”，负责把不同上游列名转换成平台内部统一任务输入。该模块应尽量做成深模块，对上层只暴露简单、稳定的记录转换接口。
- URL 下载和远程文档获取被定义为“远程文档接入模块”，职责是根据来源 URL 下载文件、标准化错误语义、输出可被文档加载链路消费的对象。该模块应复用现有远程文件下载边界，而不是在 workflow 中散落 HTTP 细节。
- 营业执照文本获取和字段抽取被定义为“营业执照文档解析模块”，职责是识别文档是否为营业执照，并抽取核心字段。V1 允许优先使用规则、Stub OCR、Stub LLM 或文本驱动方式跑通，不要求真实 OCR 或多模态模型。
- 营业执照字段模型应至少覆盖：证照类型、企业名称、统一社会信用代码、经营场所或住所、法定代表人或经营者、成立日期或营业期限起止、营业期限截止日期。V1 可接受个别非关键字段缺失，但关键字段缺失必须进入明确的失败或人工复核语义。
- 营业执照规则模块被定义为确定性规则模块，首期至少覆盖：文档类型匹配、企业名称存在与匹配、统一社会信用代码存在与匹配、有效期状态、营业执照是否缺少可审核关键字段。
- 规则结果必须统一映射到平台现有的 `RuleResult` 和 `ReviewResult` 契约，避免为营业执照新造独立输出格式。
- 营业执照审核结果持久化应保留三类信息：来源证据快照、抽取字段快照、规则结果快照。V1 使用独立 MySQL 审核结果库保存完整 `ReviewResult` 快照和营业执照投影表。
- 营业执照 use_case 应被视为一条新的单证 tracer bullet，而不是继续塞进现有 `qc_document_review` 的 `product_report` 特化逻辑里。这样更符合 README 中“多场景 use_case 骨架”的演进方向。
- 营业执照 workflow 应保持与 `food_license` 类似的节点职责：来源任务标准化、文档获取、文档类型识别、字段抽取、字段规范化、规则执行、风险汇总、人工复核路由。节点名称不要求完全一致，但边界应一致。
- 人工复核语义采用保守策略：文档类型无法识别、关键字段缺失、有效期无法解析、主体信息不足以判断时，进入 `PENDING_MANUAL_REVIEW`；明确高风险规则失败时可直接 `FAILED`。
- 多证照扩展不在本 PRD 的实现范围内，但营业执照实现必须沉淀证照类型映射、字段模型扩展方式和来源记录聚合边界，供后续烟草证接入复用。
- 并发开发时，建议按 issue 边界组织子代理：来源任务获取、远程文档与字段抽取、规则与结果落库、扩展框架梳理。各子代理共享本 PRD 的输入输出契约，不共享实现假设。

## Testing Decisions

- 好的测试应该验证外部行为，而不是节点内部实现细节。测试应关注输入什么、输出什么、状态如何变化、错误语义是否稳定，而不是 workflow 内部具体调用顺序。
- 本 PRD 对测试的基本要求是：在不依赖真实 MySQL、真实 OCR、真实 LLM、真实 OA 的情况下，完成本地可重复验证。
- 需要优先测试的模块包括：来源记录标准化、远程文档下载边界、营业执照字段抽取、营业执照规则执行、营业执照 use_case 的 `ReviewResult` 契约、审核结果落库读取。
- MySQL 取数相关测试应验证：空结果、缺字段、重复记录、非营业执照类型、URL 缺失、来源字段别名映射。
- 文档获取相关测试应验证：URL 下载成功、HTTP 错误、空文件、不支持格式、Content-Type 与扩展名冲突。
- 字段抽取相关测试应验证：标准营业执照文本、字段缺失文本、噪声 OCR 文本、非营业执照文本、日期无法解析文本。
- 规则执行相关测试应验证：规则全部通过、主体不匹配、信用代码缺失、有效期已过期、有效期无法判断、材料类型错误。
- 审核结果相关测试应验证：自动通过、自动失败、待人工复核三类主要结果；同时验证 `ReviewResult` 中摘要、风险等级、规则结果和人工复核状态的一致性。
- 结果落库相关测试应验证：审核任务可保存、再次读取结果一致、来源证据和抽取字段快照存在、规则结果快照存在。
- 并发开发时，每个子代理都应优先补“契约测试 + 行为测试”，避免多个分支同时依赖尚未稳定的内部 helper。
- 本项目现有测试可以作为 prior art 的方向包括：远程文档下载边界测试、来源记录标准化测试、use_case facade 行为测试、workflow runtime boundary 测试、结果仓库存取测试。

## Out of Scope

- 烟草专卖零售许可证解析与规则实现
- 营业执照与烟草证双证一致性比对
- OA 回写、企微或飞书通知
- 真实 OCR 服务接入
- 真实 LLM 服务接入
- 生产级 MySQL 结果仓库设计和迁移体系
- 多租户规则热加载
- 管理后台、查询页面或导出页面
- 对历史 `food_license` 主链路做大规模重构

## Further Notes

- 本 PRD 对应的当前实施顺序，应与已创建的 issue 保持一致：先完成 MySQL 拉取闭环，再完成 URL 下载与字段抽取，再完成规则与结果落库，最后整理多证照扩展边界。
- 如果后续直接使用 `tdd` skill，建议每个 issue 启动一个独立子代理，并统一遵守“先写失败测试，再补最小实现，再收口重构”的节奏。
- 建议的并发方式不是四张 issue 完全同时开工，而是采用“1 先行，2 跟进，3 提前准备测试骨架，4 最后收口”的半并发方式：
  - issue 31 先行定义来源任务输入契约；
  - issue 32 在 31 的输入契约稳定后立即并发；
  - issue 33 可以提前搭测试骨架和规则语义，但依赖 32 的字段模型定稿；
  - issue 34 最适合在 33 基本稳定后收口，不建议最早开工。
- 如果后续要从营业执照扩展到烟草证，优先复用来源记录标准化、远程文档接入、规则结果映射和结果落库边界，不应复制一套平行实现。

## Extension Boundaries for #34

- `business_license capability 只负责营业执照单证` 的识别、字段抽取、确定性规则执行和结果快照，不承担烟草证或双证一致性职责。
- 后续新增证照时，优先复用 `来源记录标准化、远程文档获取、规则执行、ReviewResult 映射、MySQL 审核结果库投影` 这些边界，而不是复制一套平行平台基础设施。
- 本 V1 不实现烟草证字段模型，不实现双证一致性规则，不实现 OA 回写，不实现企微通知。
- 后续证照应通过独立 use_case 或 capability 显式注册，避免把新证照逻辑塞进 `business_license` 单证链路。
- 跨证照继续使用统一 `ReviewResult` 顶层契约；证照专属内容放入 skill_result 和对应投影表。
