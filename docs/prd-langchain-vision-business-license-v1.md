# LangChain 多模态营业执照审核 V1 PRD

本文档定义 `document-ai-review` 使用 LangChain 接入多模态大模型完成营业执照 PDF/JPG/PNG 字段抽取与规则审核的 V1 产品范围、实现边界和测试决策。

`README.md` 仍然是项目唯一主上下文。本文档只补充营业执照从文件到结构化审核结果的多模态实现 PRD，不替代总需求文档，也不替代现有营业执照单证审核 PRD。

---

## Problem Statement

当前营业执照单证审核已经具备文本驱动的 tracer bullet：系统可以接收 OCR 文本，抽取营业执照字段，执行确定性规则，并输出统一 `ReviewResult`。但真实业务输入通常是 PDF、JPG 或 PNG 文件，审核人员和上游系统并不会提前提供 `ocr_text`。

这导致当前营业执照验证存在明显断点：

- 用户有营业执照文件，但不能直接交给系统审核；
- 扫描版 PDF、JPG、PNG 无法进入字段抽取链路；
- 传统 OCR 尚未接入，现有 OCR adapter 仍是 stub；
- 如果先要求人工 OCR，再传文本，会削弱平台自动化价值；
- 后续烟草证、食品许可证、合同图片页等场景也会重复遇到同类文件理解问题。

用户当前明确的技术方向是：使用 LangChain 作为模型调用、Prompt、结构化输出和工具封装层，后续接入 AI 大模型，优先让多模态大模型直接从营业执照 PDF/JPG/PNG 中提取结构化字段，再复用现有规则校验与审核结果契约。

## Solution

V1 建设一条“营业执照文件输入 -> 多模态大模型字段抽取 -> 字段规范化 -> 确定性规则校验 -> `ReviewResult` 输出”的可运行链路。

系统应支持审核人员或业务系统提交本地文件路径或远程文件 URL。对于 JPG/PNG 图片，系统直接将图片内容传给多模态大模型；对于 PDF，系统先判断是否存在可提取文本，若 PDF 内含可复制文字则优先走文本提取，若是扫描 PDF 或文本不足，则将 PDF 页面转换为图片后交给多模态大模型。模型输出必须经过结构化 schema 校验和字段规范化，不能直接作为最终审核结论。

LangChain 在 V1 中承担以下职责：

- 组织营业执照字段抽取 Prompt；
- 调用多模态大模型；
- 约束模型输出为结构化 JSON；
- 对模型响应做解析、重试和错误归一；
- 为 workflow 提供稳定的字段抽取接口。

营业执照合规规则由 `.agents/skills/business-license-review/SKILL.md` 维护。Python Runtime 负责读取 Skill、调用大模型执行规则、解析结构化 JSON，并由现有 `business_license` use_case、workflow、capability 和 `ReviewResult` 契约承接最终结论。

## User Stories

1. 作为 QC 审核员，我希望上传营业执照 PDF 文件后系统可以自动识别字段，以便不需要手工复制营业执照文字。
2. 作为 QC 审核员，我希望上传营业执照 JPG 图片后系统可以自动审核，以便适配手机拍照、扫描仪导出的常见材料。
3. 作为 QC 审核员，我希望上传营业执照 PNG 图片后系统可以自动审核，以便适配上游系统保存的截图或图片附件。
4. 作为 QC 审核员，我希望系统能识别扫描版 PDF，以便处理没有可复制文本的证照文件。
5. 作为 QC 审核员，我希望系统能识别电子 PDF 中的可复制文字，以便在不调用视觉模型时也能完成低成本审核。
6. 作为业务系统调用方，我希望通过统一审核 API 提交营业执照文件，以便不关心底层是 OCR、PDF 文本提取还是多模态模型。
7. 作为业务系统调用方，我希望通过远程 URL 提交营业执照文件，以便复用现有 SRM/OA 文件存储。
8. 作为业务系统调用方，我希望系统保留来源记录 ID、附件 ID 和文件 URL，以便审核结果可以追溯到原始业务记录。
9. 作为审核人员，我希望系统提取营业执照的企业名称，以便校验其是否与供应商名称一致。
10. 作为审核人员，我希望系统提取统一社会信用代码，以便校验主体身份和证照真实性基础信息。
11. 作为审核人员，我希望系统提取住所或经营场所，以便后续扩展营业执照与烟草证地址一致性校验。
12. 作为审核人员，我希望系统提取法定代表人、经营者或负责人，以便后续扩展负责人一致性校验。
13. 作为审核人员，我希望系统提取营业期限起止日期，以便判断证照是否过期或临近过期。
14. 作为审核人员，我希望系统识别长期有效的营业执照，以便不会把长期有效误判为缺失有效期。
15. 作为审核人员，我希望系统提取登记机关和发证日期，以便后续复核和报表留存。
16. 作为审核人员，我希望系统判断材料是否真的是营业执照，以便避免其他报告或证照误入营业执照规则。
17. 作为审核人员，我希望系统在模型无法确认字段时进入人工复核，以便避免不可靠字段被自动通过。
18. 作为审核人员，我希望系统展示模型抽取的字段和规则失败原因，以便人工复核时能快速定位问题。
19. 作为审核人员，我希望系统区分“模型无法读取文件”和“规则校验不通过”，以便判断是技术失败还是业务风险。
20. 作为审核人员，我希望系统对模糊、遮挡、倒置、低分辨率图片给出人工复核原因，以便材料质量问题可以被追踪。
21. 作为审核人员，我希望系统对多页 PDF 做合理处理，以便营业执照不在第一页时仍可能被识别。
22. 作为审核人员，我希望系统限制最大页数和最大文件大小，以便异常文件不会拖垮审核服务。
23. 作为平台维护者，我希望模型调用被封装为稳定 adapter，以便后续切换 OpenAI、Azure、通义、百度、私有模型或其他视觉模型时不影响规则层。
24. 作为平台维护者，我希望 Prompt 与字段 schema 位于 capability 边界内，以便营业执照、烟草证、食品许可证可以分别演进。
25. 作为平台维护者，我希望模型输出经过 Pydantic schema 校验，以便下游规则只消费稳定结构。
26. 作为平台维护者，我希望模型失败、超时、JSON 解析失败都有稳定错误码，以便 API 调用方可以做重试或人工处理。
27. 作为平台维护者，我希望模型抽取结果、模型元数据和来源证据一起保存，以便后续审计留痕和问题排查。
28. 作为平台维护者，我希望系统记录模型供应商、模型名、调用耗时和重试次数，以便评估成本与稳定性。
29. 作为平台维护者，我希望大模型只负责字段抽取，不负责最终审核结论，以便业务规则保持可解释、可测试、可追溯。
30. 作为平台维护者，我希望当前文本驱动的 `ocr_text` 入口继续可用，以便已有测试和调试方式不被破坏。
31. 作为开发人员，我希望新增文件输入链路遵循现有 `ReviewService -> use_case -> workflow -> capability -> rules -> ReviewResult` 架构，以便不破坏平台主线。
32. 作为开发人员，我希望本地测试可以使用 fake multimodal LLM adapter，以便不依赖真实模型 API。
33. 作为开发人员，我希望集成测试可以覆盖 PDF、JPG、PNG 三类输入，以便确认文件路由和错误语义稳定。
34. 作为开发人员，我希望模型 Prompt 可以单独测试结构化输出约束，以便减少线上 JSON 解析失败。
35. 作为后续烟草证开发者，我希望营业执照视觉抽取链路沉淀可复用的文档页渲染、图片输入和模型 adapter，以便烟草证接入不重复建设基础设施。

## Implementation Decisions

- V1 以营业执照单证为唯一业务目标，不同时实现烟草证或双证一致性比对。
- V1 的核心路线是多模态大模型字段抽取，不以传统 OCR 作为主路径。传统 OCR 可以作为未来可选 adapter，但不阻塞本 PRD。
- LangChain 用于模型调用、Prompt 管理、结构化输出解析和模型 adapter 封装；LangGraph 继续用于 workflow 编排。
- 大模型只输出结构化字段和模型侧置信信息，不直接输出最终审核通过或不通过。
- 最终审核结论继续由确定性规则模块生成，并映射到统一 `RuleResult` 和 `ReviewResult`。
- 现有 `ocr_text` 输入继续保留，作为调试入口、测试入口和上游已完成 OCR 时的低成本入口。
- 新增文件输入支持本地文件路径和远程文件 URL 两类来源。远程文件下载应复用现有远程文档接入边界。
- 文件类型 V1 支持 PDF、JPG、JPEG、PNG。其他格式返回稳定的不支持格式错误或进入人工复核，不做隐式转换。
- PDF 处理采用两段策略：先尝试提取 PDF 内嵌文本；当文本为空、过短或不含营业执照特征时，再将页面渲染为图片交给视觉模型。
- PDF 页面渲染应封装为深模块，对 workflow 暴露简单接口，例如“输入文件，输出可供视觉模型消费的页面图片集合和元数据”。
- JPG/PNG 输入应进行基础读取和大小校验，并封装为统一的视觉模型输入对象。
- 多页 PDF V1 可设置最大处理页数。默认优先处理前若干页，并在元数据中记录实际处理页码。
- 文件大小、页数、图片分辨率应有上限。超过限制时返回稳定错误或进入人工复核，避免模型调用成本失控。
- 营业执照 capability 应新增视觉抽取器边界，统一处理文本输入和视觉输入后的字段 schema。
- 营业执照字段 schema 至少包括：证照类型、企业名称、统一社会信用代码、住所或经营场所、法定代表人或经营者、营业期限起始日期、营业期限截止日期、登记机关、发证日期、字段置信信息。
- 长期有效应规范化为既有规则可识别的稳定值，不应让模型自由输出多种表达。
- 模型输出必须经过 Pydantic schema 校验。字段缺失、类型错误、日期无法规范化、JSON 解析失败都应进入明确错误语义。
- Prompt 应要求模型只根据文件内容抽取，不得补全、猜测或编造字段。无法识别时输出空值和原因。
- Prompt 应要求模型返回原始字段文本和规范化字段值，以便人工复核可以看到模型依据。
- 视觉模型 adapter 应支持 fake 实现和真实实现。fake 实现用于单元测试，真实实现通过配置选择模型供应商和模型名称。
- 模型配置应通过 settings 或环境变量注入，包括 API key、base URL、model name、timeout、max retries 等。
- 缺少模型配置时，系统应返回稳定错误或进入人工复核，不能静默输出空字段并伪装成业务审核失败。
- 模型调用错误应区分认证失败、超时、限流、模型不可用、响应格式错误、文件预处理失败。
- workflow 中的 `load_document` 节点应从“只接文本或 stub”扩展为“解析输入来源并产出文本、视觉输入或错误状态”。
- workflow 中的字段抽取节点应根据输入类型选择文本抽取器或视觉抽取器。
- 规则执行节点不应关心字段来自 OCR、PDF 文本还是视觉模型，只消费规范化后的营业执照字段。
- `ReviewResult.skill_result` 应保留文件输入摘要、模型抽取元数据、原始字段快照、规范化字段快照、来源证据和错误详情。
- MySQL 营业执照投影应继续保存抽取字段、规则结果和来源证据，并补充模型调用元数据字段或 JSON 快照。
- FastAPI 应新增营业执照审核接口，避免复用食品许可证路径。接口应调用 `ReviewService.review(..., use_case_name="business_license")`。
- API 请求应避免同时提交 `ocr_text` 和文件输入；二者同时存在时返回稳定的歧义输入错误。
- API 返回应使用现有 `ReviewResult` JSON 形态，避免为营业执照视觉链路单独创建不兼容输出。
- 后续烟草证可复用文件输入、PDF 渲染、视觉模型 adapter 和错误语义，但烟草证字段 schema 与规则不放入本 V1。

## Testing Decisions

- 好的测试应验证外部行为和契约，不应绑定 LangChain 内部调用细节、Prompt 具体措辞或 LangGraph 节点内部顺序。
- 单元测试应覆盖文件输入校验：空输入、同时提交文本和文件、不支持格式、文件不存在、文件过大、PDF 页数超限。
- 单元测试应覆盖 PDF 文本提取路径：可复制文字 PDF 应不调用视觉模型即可完成营业执照字段抽取。
- 单元测试应覆盖扫描 PDF 路径：PDF 无可提取文本时应进入页面渲染和视觉模型抽取。
- 单元测试应覆盖 JPG、JPEG、PNG 图片路径：图片输入应能通过 fake multimodal adapter 返回结构化字段。
- 单元测试应覆盖模型输出解析：合法 JSON、缺字段 JSON、日期格式异常、非 JSON 响应、模型拒答、空响应。
- 单元测试应覆盖字段规范化：中文日期、ISO 日期、长期有效、空值、带空格的统一社会信用代码。
- 规则测试应继续覆盖营业执照类型匹配、主体名称匹配、统一社会信用代码匹配、有效期状态和关键字段完整性。
- Use case 测试应覆盖三类最终状态：自动通过、待人工复核、审核失败或高风险未通过。
- API 测试应覆盖营业执照 HTTP 入口的请求校验、成功响应和稳定错误响应。
- 持久化测试应覆盖模型元数据、文件来源、抽取字段、规范化字段、规则结果和人工复核原因的保存读取。
- fake multimodal LLM adapter 应成为默认测试依赖，所有 CI 测试不得调用真实大模型 API。
- 可选的真实模型 smoke test 应通过环境变量显式开启，默认跳过，避免测试成本和密钥泄漏风险。
- 现有营业执照文本驱动测试必须继续通过，确保新增视觉能力不破坏 `ocr_text` 调试路径。
- 可参考现有测试方向：营业执照 extractor、营业执照 rules、营业执照 use_case、营业执照 persistence、食品许可证本地 PDF 输入边界、远程文档下载边界。

## Out of Scope

- 烟草专卖零售许可证字段抽取与规则实现。
- 营业执照与烟草证双证一致性比对。
- 生产级 OA 回写、企微或飞书通知。
- 管理后台、人工复核页面、查询导出页面。
- 传统 OCR 引擎私有化部署。
- 模型供应商评测平台。
- 多租户模型路由和租户级 Prompt 配置。
- 对合同审核或批次报告审核做视觉模型改造。
- 让大模型直接裁决最终审核结论。
- 为所有 PDF 复杂版式提供完整文档理解能力，V1 只保证营业执照审核场景。

## Further Notes

- 当前代码已经有营业执照文本驱动闭环，新增能力应优先复用既有字段 schema、规则模块、use_case 和 MySQL 审核结果库投影，不应重写一条平行链路。
- 本 PRD 与现有 V1 架构不冲突：FastAPI 仍是 HTTP 边界，Review Service 仍是任务入口，LangGraph 仍是 workflow 编排，LangChain 负责模型调用和结构化输出，Python 规则引擎负责确定性审核。
- 本 PRD 与 `docs/prd-business-license-review-v1.md` 的关系是递进关系：后者定义营业执照单证审核 tracer bullet，本文档定义其文件输入和多模态大模型提取能力。
- 推荐实施顺序：
  1. 先补营业执照 HTTP API 和文件输入校验。
  2. 再抽象 PDF/图片预处理与视觉模型输入对象。
  3. 再实现 fake multimodal adapter 和结构化输出解析。
  4. 再接真实 LangChain 多模态模型配置。
  5. 最后补持久化元数据和真实模型 smoke test。
- 如果后续要创建 GitHub Issue，建议标题为：`feat: 支持 LangChain 多模态营业执照文件抽取审核`，并添加 `ready-for-agent` label。
